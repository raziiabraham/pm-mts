# Chapter 3: Delivery Is a Product Skill

## Product question

> What path must this change travel before users receive it, and what evidence should exist at each transition?

## What the agent brings into chat

1. [Commit-to-production delivery figure](../figures/png/figure_3_1.png)
2. [Feature Release Path Record](../templates/release_path_record.md)

No terminal is required to complete the record.

## Five-minute walkthrough

1. State the product objective and smallest useful slice.
2. Map the expected branch, pull request, automated checks, local environment, preview or staging environment, review, and production path.
3. Add happy, unhappy, permission, and provider-failure verification.
4. Decide whether the release needs a flag, staged rollout, monitoring, rollback, or disable path.
5. Name the evidence required before the team can say “built,” “deployed,” “exposed,” and “learned.”

## Take it to your product

Complete one release path for a feature currently being discussed. Bring the unresolved checks and release controls into planning before implementation begins.

## Ask an AI coding agent

> Read this guide and the Feature Release Path Record. Ask me for one feature idea and the team’s actual delivery environments. Help me map the path from branch to production, including failure tests, review, flags, rollout, monitoring, and rollback. Do not invent CI checks or environments; mark unknowns for engineering confirmation.

## Evidence boundary

A release plan is not proof that a change passed checks, reached users, or created value. Those claims require records from the real delivery system and post-release evidence.
