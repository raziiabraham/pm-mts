# Chapter 11: The Moment Traditional PM Stops Being Enough

## Product question

> Is the current system failing because of missing rules and workflow—or because the product must handle genuine uncertainty?

## Start with the visible artifacts

1. [Deterministic-probabilistic boundary figure](../figures/png/figure_11_1.png)

## Five-minute walkthrough

1. Describe the problem without proposing AI.
2. Identify what the current deterministic system assumes about sources, entities, offerings, and synthesis.
3. Separate source uncertainty, entity uncertainty, and synthesis uncertainty.
4. Define the user decision the capability must improve.
5. Draw which behavior should remain deterministic and where bounded judgment may be useful.

## Take it to your product

Create a problem-shape evidence packet: one evidence card, one state map, the decision to improve, the deterministic/probabilistic boundary, and the unknowns that must be resolved before solution selection.

## Ask an AI coding agent

> Read this guide, the Evidence Card, and API State Worksheet. Help me map a failing product behavior without starting from AI. Separate verified facts, source uncertainty, entity uncertainty, synthesis uncertainty, and unsupported interpretation. Ask which decisions can remain rules-based and what evidence would justify probabilistic judgment.

## Evidence boundary

Recognizing uncertainty does not prove that AI is needed. The packet should make no-build, workflow, data-quality, deterministic, and specialist options easier to compare in Chapter 12.
