## Five-minute walkthrough

1. Begin with a journey question and decision, not a generic request for “tracking.”
2. Separate system-of-record entities from behavioral events.
3. Define each event’s trigger, actor, object, required properties, and owner.
4. Connect event definitions to metrics and state what those metrics cannot establish.
5. Inspect valid and invalid envelopes before offering the optional check.

## Take it to your product

Write one tracking plan that another person can implement and QA without guessing what the behavior means. Include ownership, privacy, identity, and data-quality questions.

## Ask an AI coding agent

> Read this guide, `companion/analytics/README.md`, and `tracking_plan.json`. Explain the product decision behind each event. Show me one valid and one invalid envelope and ask me to predict which should pass. Then interview me through the Tracking Plan for one real journey. Do not treat an event name as evidence that production tracking works.

## Evidence boundary

A tracking plan defines intended semantics. It does not prove the event fires correctly, identities resolve properly, dashboards calculate the metric faithfully, or the observed behavior caused an outcome.
