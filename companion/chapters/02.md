# Chapter 2: APIs Are Where Product Ideas Become System Behavior

## Product question

> Does the API contract expose every success, failure, recovery, data, and ownership state the product experience needs?

## Five-minute walkthrough

1. Read the host, method, path, and request body.
2. Predict the interface behavior for success, invalid input, and source-provider failure.
3. Compare your prediction with the three response bodies.
4. Map each state to user feedback, retry, operations visibility, analytics, and human control.
5. Record the open security, permission, retention, and ownership questions.

## Take it to your product

Choose one user action and complete an API State Worksheet. The output should let product, engineering, and design disagree about the same visible states instead of carrying separate assumptions into build.

## Evidence boundary

The host is explicitly fictional and non-live. A passing check proves only that the supplied teaching files preserve the declared contract; it does not prove production behavior, security, reliability, or user value.
